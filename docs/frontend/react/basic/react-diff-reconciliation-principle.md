# React Diff / Reconciliation 原理

Diff 是 React 在两棵虚拟 DOM 树之间寻找最小变更的过程。

## React 的核心假设（面试高频）

为了把复杂问题控制在可接受复杂度，React 采用两个重要假设：

1. 不同类型节点会生成不同子树（直接替换）。
2. 同层节点通过 `key` 标识稳定身份。

这使得常见场景下 diff 不必走昂贵的全量树编辑距离算法。

## 同层比较的典型规则

### 1) 节点类型不同

直接卸载旧节点，挂载新节点。

### 2) 节点类型相同

复用节点，比较属性差异并更新。

### 3) 列表节点

- 有稳定 key：可识别移动/复用关系，更新更准确。
- key 不稳定（如随机数）：会导致大量重建。
- 用 index 作 key 在插入/删除/重排时容易状态错位。

## 为什么 key 很关键

`key` 是列表元素的“身份 ID”，不是给组件读的业务数据。它决定 React 是否复用某个节点及其内部状态。

## 结合 Fiber 看 Diff

在 Fiber 架构下，reconciliation 可以拆分为多个工作单元，按优先级调度，必要时让步给高优任务（例如用户输入），再继续完成。

## 面试常见追问

### Q1：Diff 是不是全树逐节点暴力比较？

不是。React 主要按同层规则与 key 做启发式比较，避免高复杂度算法。

### Q2：为什么 key 不能用随机值？

每次 render 都变化会让 React 认为节点全是新节点，失去复用，性能和状态都可能出问题。

## 30 秒口述模板

React Diff 是在新旧虚拟 DOM 之间找最小变更。它依赖“同类型可复用、不同类型替换、列表靠 key 识别身份”这套启发式规则来降低复杂度。Fiber 让这个过程可调度、可中断，从而在复杂页面里保持交互响应。
